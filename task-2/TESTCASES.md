## Общая информация
Тестируются ручки микросервиса:
1. Создание объявления
2. Получение объявления по идентификатору
3. Получение всех объявлений по sellerId
4. Получение статистики по itemId

## Позитивные сценарии

### TC-001 Создание объявления с валидными данными
**Тип:** Positive  
**Предусловия:** Подготовлен уникальный sellerId в диапазоне 111111–999999.  
**Шаги:**
1. Отправить POST-запрос на создание объявления с валидным телом.
2. Проверить код ответа.
3. Проверить тело ответа.
**Ожидаемый результат:**
- Возвращается успешный статус-код.
- В ответе присутствует уникальный идентификатор объявления.
- Созданное объявление доступно для последующего получения по id.

### TC-002 Получение объявления по существующему id
**Тип:** Positive  
**Предусловия:** Есть ранее созданное объявление.
**Шаги:**
1. Отправить GET-запрос по id объявления.
2. Проверить код ответа.
3. Сравнить данные ответа с данными при создании.
**Ожидаемый результат:**
- Возвращается успешный статус-код.
- Поля объявления соответствуют сохранённым значениям.

### TC-003 Получение списка объявлений по sellerId
**Тип:** Positive  
**Предусловия:** Создано одно или несколько объявлений с одинаковым sellerId.
**Шаги:**
1. Отправить GET-запрос по sellerID.
2. Проверить код ответа.
3. Проверить, что в списке есть созданные объявления.
**Ожидаемый результат:**
- Возвращается успешный статус-код.
- В ответе содержатся объявления указанного продавца.

### TC-004 Получение статистики по существующему itemId
**Тип:** Positive  
**Предусловия:** Создано объявление.
**Шаги:**
1. Отправить запрос на получение статистики по itemId.
2. Проверить код ответа.
3. Проверить структуру статистики.
**Ожидаемый результат:**
- Возвращается успешный статус-код.
- В ответе присутствуют поля статистики.

## Граничные и corner-cases

### TC-016 Проверка поведения при повторной отправке одинакового POST-запроса
**Тип:** Corner case  
**Шаги:**
1. Дважды отправить POST-запрос с одинаковым телом.
**Ожидаемый результат:**
- Создаются разные сущности с разными id, если идемпотентность не предусмотрена.
- Если сервис реализует идемпотентное поведение, это долно быть явно задокументировано.

### TC-017 Создание объявления с минимально разумно валидными значениями
**Тип:** Boundary  
**Шаги:**
1. Отправить POST-запрос с минимально разумно валидными значениями: name длиной 1 символ, price = 1, statistics со значениями 0 и валидный sellerID.
**Ожидаемый результат:**
- Объявление успешно создаётся.

### TC-018 Создание объявления с большими значениями полей при отсутствии явно задокументированных ограничений
**Тип:** Boundary  
**Шаги:**
1. Отправить POST-запрос с длинным name, большим значением price и валидным sellerID.
**Ожидаемый результат:**
- Сервис либо принимает значение, либо корректно возвращает ошибку валидации, но не должен выдвать ошибку 500.

### TC-019 sellerId на границах рекомендованного диапазона
**Тип:** Boundary  
**Шаги:**
1. Создать объявления с sellerId = 111111 и sellerId = 999999.
**Ожидаемый результат:**
- Объявления успешно создаются.


## Нефункциональные проверки

### TC-020 Время ответа ручки создания объявления
**Тип:** Non-functional  
**Шаги:**
1. Отправить запрос на создание объявления.
2. Замерить время ответа.
**Ожидаемый результат:**
- Время ответа не должно превышать заранее установленный порог, например 5 секунд для тестового стенда.

### TC-021 Корректность Content-Type
**Тип:** Non-functional  
**Шаги:**
1. Выполнить запрос.
2. Проверить заголовки ответа.
**Ожидаемый результат:**
- Возвращается корректный `Content-Type`, например `application/json`.

### TC-022 Контракт ответа
**Тип:** Non-functional / Contract  
**Шаги:**
1. Выполнить запрос к каждой ручке.
2. Проверить типы полей и обязательные поля ответа.
**Ожидаемый результат:**
- Структура ответа соответствует ожиданиям.

### TC-023 Проверка формата поля createdAt в ответах API
**Тип:** Non-functional / Contract
**Предусловия:** Есть ранее созданное объявление.
**Шаги:**
1. Выполнить запрос к ручкам, возвращающим данные объявления.
2. Проверить наличие поля createdAt в ответе.
3. Проверить, что createdAt является строкой.
4. Проверить, что значение соответствует корректному формату даты и времени.
**Ожидаемый результат:**
- Поле createdAt присутствует в ответе.
- Поле createdAt имеет строковый тип.
- Значение createdAt соответствует корректному формату даты/времени.
